# Notes for future plans

## Merge Static Rule Blocks — after Phase 1 (Core merge engine + persisted schema)

- Merge entry point: `D4LootBench.Core.Progression.ProgressionFilterMerger.Merge(overrideBlock, betterGear, overriddenByBlock, filterName)` → `MergedFilterResult { Ruleset, Warnings, OverBudget }`. Blocks are `IReadOnlyList<FilterRule>`; `betterGear` is the generator's `FilterRuleset`. Reason: keeps ordering/first-match-wins logic in one testable place.
- Static blocks persist on `ProgressionProfile` as **base64 share codes** (`OverrideBlockCode`, `OverriddenByBlockCode`), not rule-JSON. Reason: avoids a `FilterDataContext` dependency in `ProfileSerializer`; blocks are imported *from* codes. Phase 2/3 wizard must `FilterCodec.Decode(...)` a stored code into rules before calling `Merge`.
- Profile schema is now **v2**; v1 files deserialize (missing keys default to null via STJ positional-record defaults). Don't reorder `StoredProfile` params; append only, with `= null` defaults.
- The merger sanitizes both static blocks itself — override drops *all* Hide-Alls; overridden-by drops *interior* Hide-Alls but keeps a *trailing* one as the final catch-all. So the wizard need not pre-strip Hide-Alls. Output invariant: exactly one trailing `Visibility.HideAll`.
- 25-rule cap: better-gear is trimmed from the tail first (never static rules). `OverBudget == true` means the static blocks alone exceed 25 → the caller must surface a validation error, not just a warning. Always surface `Warnings` to the user.

## Merge Static Rule Blocks — after Phase 2 (wizard wiring: import, merge, persist)

- The wizard holds each block as its **base64 code** in `[ObservableProperty] OverrideBlockCode` / `OverriddenByBlockCode` (empty string = no block), never as a decoded rule model. Derived `OverrideRuleCount` / `OverriddenByRuleCount` decode-on-read (0 when empty/undecodable) and are refreshed by the code-changed partial hooks. Reason: the block *is* an imported share code; keep one source of truth. Phase 3's **Edit…** button must round-trip through the code string (decode → edit → `FilterCodec.Encode` → set the code), not add a parallel rule field.
- Import/edit are non-destructive: `ImportBlockFromClipboard` reads the `_getClipboard` seam, validates via `FilterCodec.Decode` in try/catch, and on any failure leaves the block untouched + sets an error status. Phase 3 Edit should follow the same "never clobber a good block on failure" rule.
- `Generate` merges at export time: `_merger.Merge(DecodeBlock(override), filter.Ruleset, DecodeBlock(overriddenBy), "Progression Filter")`; `ShareCode = FilterCodec.Encode(merged.Ruleset)`. Merge warnings **and** `merged.Ruleset.Validate()` errors both land in `Warnings`; a non-empty validation set sets error status but still advances to Result + auto-saves. `DecodeBlock` throws only on a corrupt stored code, caught by Generate's outer `catch`.
- `StaticRules` enum step sits **between `Goal` and `Result`**; the **Generate** button lives there (Goal's button became **Next: Static Rules**). Step headers are "N of 5". The two XAML block cards carry `<!-- Edit… button added in Phase 3 -->` placeholders — that's where Phase 3 wires the editor.
- New VM ctor params: `ProgressionFilterMerger merger` (after `generator`) and `Func<string>? getClipboard = null` (after `setClipboard`). `ProgressionFilterMerger` is a DI singleton. Test factories pass `setClipboard` positionally and `confirm:` by name, so appending params before `confirm` stays source-compatible — keep that ordering if you extend the ctor again.

## Merge Static Rule Blocks — after Phase 3 (per-block visual editor dialog)

- The manual "Edit…" capability is a VM seam `Func<FilterRuleset, string, string?> _editBlock` (input: block ruleset + dialog title; output: edited block's base64 code, or `null` on Cancel), defaulted to `(_, _) => null` for headless/test construction. Ctor param `editBlock` sits **before** `confirm` (Phase 2's ordering rule still holds: append before `confirm`, tests pass it by name). Reason: the VM must never `new` up a WPF `Window` — the production seam (registered in `ServiceConfiguration`) owns building `VisualEditorViewModel` + showing `BlockEditorWindow` and the encode-on-OK contract.
- `EditBlock` decodes the current block code inside a **try/catch** (like `ImportBlockFromClipboard`/`CountRules`) — a stored code can reach it unvalidated (e.g. `OpenSelectedProfile` restores `OverrideBlockCode`/`OverriddenByBlockCode` verbatim from persisted profile JSON). Any future code that decodes a stored block code must guard the same way; a corrupt code sets an error status and never opens the editor / clobbers the block. Blank block → a **named** empty `FilterRuleset` ("Override Rules" / "Overridden-By Rules").
- `BlockEditorWindow` (Views) is a thin modal OK/Cancel shell hosting the shared `VisualEditorView` (its `DataContext` is a `VisualEditorViewModel`; OK sets `DialogResult=true`). It is intentionally **not** unit-tested headlessly — the seam is faked in tests; the window is manual-smoke only. Reuse this pattern if another feature needs to host the rule editor modally.
